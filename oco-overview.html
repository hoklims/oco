La progression

Q0 → Q11 : Phase par phase

Chaque phase a resolu un probleme precis et ajoute une brique durable.

Q0
Fondations
Stack polyglot + orchestration deterministe
Rust pour le noyau (perf, FTS5, Tree-sitter), Python pour le ML optionnel, TypeScript pour VS Code. Modele a 5 actions avec routage 100% deterministe — zero appel LLM pour les decisions de politique.
ADR-001 ADR-002 ADR-003 11 crates ~110 tests
Le chassis de la voiture : moteur, direction, freins.
Q1
Verification & Raisonnement
La verification devient serieuse, la memoire s'organise
Probleme : Un booleen has_verified ment. Modifier un fichier apres verification ne declenchait rien. Et le buffer perdait les decouvertes apres compression.
Solution : VerificationState par fichier avec freshness + WorkingMemory structuree (findings, hypotheses, faits verifies). Validation empirique des hooks Claude Code.
ADR-004 ADR-005 ADR-007 verification.rs memory.rs
Du post-it "verifie" au cahier de labo avec dates et niveau de confiance.
Q2
Orchestration emergente
Les plans deviennent des DAGs dynamiques
Probleme : Plans plats, pas de parallelisme, templates rigides.
Solution : ExecutionPlan comme DAG, deux planners (Direct pour Trivial/Low, LLM pour Medium+), GraphRunner avec execution parallele, LlmRouter multi-modele, CapabilityRegistry unifie, Agent Teams (Mesh/HubSpoke/Pipeline).
ADR-008 plan.rs capability.rs team.rs planner/ graph_runner.rs
D'une to-do list plate a un chef d'orchestre qui compose en temps reel.
Q3
Trust & Policy Packs
La confiance devient un signal gradue
Probleme : Pas de profil par repo, verification hardcodee, pas de verdict de confiance.
Solution : RepoProfile auto-detecte, PolicyPack (Fast/Balanced/Strict), TrustVerdict structure dans RunSummary.
ADR-006 profile.rs PolicyPack TrustVerdict
Le systeme s'adapte au terrain au lieu d'appliquer une politique aveugle.
Q4
Continuite & Mission Memory
La memoire traverse les sessions
Probleme : La compaction efface l'etat. Chaque oco run repart de zero.
Solution : CompactSnapshot pour survivre aux compactions intra-session. MissionMemory comme artefact durable inter-session (faits, hypotheses, plan, verification, trust verdict). Persiste dans .oco/runs/<id>/mission.json.
ADR-009 ADR-010 mission.rs CompactSnapshot
Le carnet de transmission entre deux equipes de nuit.
Q5
Scorecard multi-dimensionnelle
7 dimensions pour evaluer un run
Probleme : Pass/fail binaire, pas de detail, pas de regression detection.
Solution : RunScorecard avec 7 dimensions ponderees (Success ×3, TrustVerdict ×2, VerificationCoverage ×1.5, MissionContinuity ×1, CostEfficiency ×1, ErrorRate ×1, ReplanStability ×0.5). ScorecardComparison pairwise + BatchComparison.
ADR-011 scorecard.rs ScorecardBuilder oco eval-compare
Du "recu/recale" au bulletin de notes avec commentaires du prof.
Q6
Quality Gates
Pass / Warn / Fail pour la CI
Probleme : Les comparaisons montrent des deltas mais pas de verdict actionnable.
Solution : GateVerdict (exit 0/1/2), GatePolicy avec 3 presets (Strict/Balanced/Lenient), GateResult::evaluate() pure et deterministe. EvalBaseline pour les snapshots.
ADR-012 gate.rs oco eval-gate
Le controle technique : la voiture passe ou ne passe pas.
Q7
Config repo des gates
Le contrat de qualite vit dans le repo
Section [gate] dans oco.toml : baseline_path, default_policy, overrides de seuils. oco eval-gate fonctionne sans argument si la config existe.
ADR-013 GateConfig oco.toml [gate]
Q8
Fraicheur des baselines
Les baselines perimees sont detectees
BaselineFreshness (Fresh/Aging/Stale/Unknown) avec seuils configurables (defaut : 14j fresh, 30j stale). GateReviewArtifact combinant gate + freshness + recommandations en Markdown et JSON.
ADR-014 BaselineFreshness fresh_days / stale_days
Q9
Review Packet unifie
Un seul document pour decider
ReviewPacket = scorecard + gate + mission + freshness. MergeReadiness : Ready / ConditionallyReady / NotReady / Unknown. 3 formats de sortie (terminal / markdown / json).
ADR-015 review_packet.rs oco runs review-pack
Le dossier complet du patient remis au medecin, pas 5 feuilles volantes.
Q10
Config review
Le review packet se configure par repo
Section [review] dans oco.toml : auto_save, default_format (terminal/markdown/json), output_dir optionnel. Pas d'ADR dedie — extension naturelle de Q9.
ReviewConfig oco.toml [review]
Q11
Baseline Promotion & Audit Trail
Chaque changement de baseline est trace et justifie
PromotionRecommendation (Promote/Review/Reject) calcule depuis gate + freshness. BaselineDiffSummary par dimension. PromotionRecord durable. BaselineHistory append-only. Reject bloque par defaut, --force documente dans l'historique.
ADR-016 PromotionRecord BaselineHistory oco baseline-promote
Le registre notarie : chaque transfert signe, date, archive.

En pratique

Le parcours d'un run moderne

Ce qui se passe concretement quand tu lances oco run aujourd'hui.

1

Classification

Le policy engine classifie ta requete (Trivial / Low / Medium / High) par pattern matching deterministe. Zero appel LLM.

2

Profiling du repo

Detection automatique du stack (Cargo.toml → Rust), sensitive paths, policy pack. OCO s'adapte au terrain.

3

Plan emergent

Le LlmPlanner genere un DAG unique : steps avec dependances, roles, modeles et effort par step. Le plan emerge du contexte, pas d'un template.

4

Execution parallele

Le GraphRunner lance les steps prets en parallele (tokio::spawn). Budget pre-reserve par batch. CancellationToken pour chaque step.

5

Verification automatique

Apres chaque step d'implementation (verify_after: true), le verifier lance build + clippy + tests. En cas d'echec, replan (max 3 tentatives).

6

Memoire & Mission

WorkingMemory accumule findings, hypotheses, faits. En fin de run, MissionMemory construite et persistee.

→ mission.json
7

Scoring 7 dimensions

ScorecardBuilder calcule Success, TrustVerdict, VerificationCoverage, MissionContinuity, CostEfficiency, ErrorRate, ReplanStability → score composite.

→ scorecard.json
8

Gate (optionnel)

Compare scorecard vs baseline avec la politique du repo. Verdict : Pass (exit 0) / Warn (exit 1) / Fail (exit 2).

→ gate-report.md/json
9

Review Packet

Bundle unique : scorecard + gate + mission + freshness → MergeReadiness. Un seul document pour decider.

→ review-packet.md/json
10

Promotion (optionnel)

Si satisfait : promouvoir la scorecard en nouvelle baseline. Backup atomique, diff par dimension, record dans l'audit trail.

→ baseline.json + baseline-history.json

A surveiller

Zones encore fragiles

Risques residuels observes depuis le repo. Pas dramatique, mais a garder en tete.

FERME — AGENTS.md desynchronise

Reecrit pour refleter les 14 crates, 1000+ tests, 4 providers, Q2-Q11, .claude/ skills.

FERME — Poids de scorecard non configurables

ScorecardWeights dans [scorecard] de oco.toml. Branche dans tous les chemins runtime : oco run, review-pack, eval-gate, baseline-save, baseline-promote, runs compare. Fallback strict sur les poids par defaut. 12 tests.

PARTIEL — Hooks non valides en conditions adverses

Tests adversaires ajoutes (payloads malformes, auth edge cases, champs inattendus). Sessions longues et agents concurrents restent non testes.

FERME — BaselineHistory non bornee

BaselineHistory::prune(keep) + CLI oco baseline-history-prune --keep N --dry-run. 6 tests.

FERME — MergeReadiness non configurable

MergeReadinessConfig dans [review.merge_readiness] : 3 toggles booleens. Branche dans ReviewPacket::build_with_config() et review-pack CLI. Blocages durs non configurables. 8 tests.

FERME — Dedup MissionMemory par hash textuel

normalize_for_dedup() : trim, collapse whitespace, lowercase. Dedup deterministe sans semantique floue. Tests de non-regression.

FERME — Sub-plan max depth hardcode (2)

Constante nommee MAX_SUB_PLAN_DEPTH = 2, explicitement fixe. ADR-008, code, et tests alignes.

Conclusion executive

Au depart, OCO etait un orchestrateur fonctionnel mais amnesique : boucle plate, verification booleenne, zero memoire inter-session, pas de scoring, pas de gates.

Aujourd'hui, c'est un systeme de qualite continue pour l'orchestration IA. Chaque run produit un plan emergent (DAG), accumule des connaissances (mission memory), est evalue sur 7 dimensions (scorecard), passe des gates CI-ready, genere un review packet unifie, et peut promouvoir sa baseline avec audit trail complet.

Le saut : de 110 a 1000+ tests, de 11 a 14 crates, de 7 a 16 ADRs, de 6 a 20+ commandes CLI. Mais le vrai changement est architectural : OCO est passe d'un executeur de taches a un systeme qui mesure, juge, trace et audite la qualite de son propre travail.
